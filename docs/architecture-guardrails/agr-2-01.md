# AGR-2.01 - Semantic Data Model Changes

| Status | Created     | Post-History                    |
|--------|-------------|---------------------------------|
| Active | 20-Jan-2026 | Initial creation                |

## Why

Semantic data models in the [Tractus-X Semantic Models Repository](https://github.com/eclipse-tractusx/sldt-semantic-models) are the foundation for data exchange in the Catena-X ecosystem. These models define the structure, semantics, and constraints of data shared between participants.

Changes to semantic models have far-reaching impacts:

- **Breaking changes** can disrupt existing integrations
- **Version conflicts** can cause interoperability issues  
- **Inconsistent semantics** reduce data quality
- **Governance gaps** lead to uncontrolled model proliferation

This guardrail ensures that semantic model changes are:
- Properly governed and reviewed
- Backward compatible when possible
- Well-documented and versioned
- Aligned with ecosystem needs

## Description

This guardrail defines **when** and **how** semantic data models in the sldt-semantic-models repository must be created, modified, or deprecated.

### Key Requirements

:::danger MUST
Any change to a semantic model **must**:
- Follow the semantic versioning scheme (MAJOR.MINOR.PATCH)
- Be proposed through a GitHub issue in the sldt-semantic-models repository
- Include a clear rationale and impact analysis
- Be reviewed and approved by the data modeling working group
- Include updated documentation and examples
- Maintain backward compatibility for MINOR and PATCH versions
:::

:::warning SHOULD
New semantic models **should**:
- Reuse existing models where possible
- Follow Catena-X naming conventions
- Include comprehensive documentation
- Provide usage examples
- Consider future extensibility
:::

## Scope

### In Scope
- Creating new aspect models
- Modifying existing aspect models
- Deprecating aspect models
- Versioning strategy
- Review and approval process

### Out of Scope
- Ontology changes (see [AGR-3.01](./agr-3-01.md))
- API specifications (covered in specific standards)
- Implementation-specific data transformations

## When to Create or Change a Semantic Model

### Create a New Model When:

1. **New Use Case Requirements**
   - A new use case requires data structures not covered by existing models
   - Existing models cannot be extended to meet the requirements
   - The new model addresses a distinct business domain

2. **Breaking Changes Required**
   - Changes to existing models would break backward compatibility
   - A new major version is warranted

3. **Domain Separation**
   - Clear separation from existing domains
   - Different governance or lifecycle

:::tip Best Practice
Before creating a new model, thoroughly search existing models. Reuse and extension are preferred over duplication.
:::

### Modify an Existing Model When:

1. **Minor Enhancements** (MINOR version bump)
   - Adding optional properties
   - Extending enumerations with new values
   - Adding new optional constraints
   - Improving documentation

2. **Bug Fixes** (PATCH version bump)
   - Correcting documentation errors
   - Fixing typos in descriptions
   - Clarifying ambiguous definitions
   - Adding missing examples

3. **Breaking Changes** (MAJOR version bump)
   - Removing or renaming properties
   - Changing property types
   - Making optional properties mandatory
   - Changing validation constraints
   - Removing enumeration values

:::danger Breaking Changes
Breaking changes require a new MAJOR version and a deprecation plan for the old version.
:::

## Implementation Guidelines

### Step 1: Assess the Need

Before making changes, answer these questions:

- Is there an existing model that can be reused or extended?
- What is the business justification for this change?
- Who are the affected stakeholders?
- What is the impact on existing implementations?

### Step 2: Create a Proposal

Submit a GitHub issue in the sldt-semantic-models repository with:

```markdown
## Model Change Proposal

**Type**: [New Model / Modification / Deprecation]

**Model Name**: [Name and version]

**Rationale**: 
[Explain why this change is needed]

**Business Context**:
[Describe the use case and business need]

**Impact Analysis**:
- Affected stakeholders: [List]
- Breaking changes: [Yes/No - details if yes]
- Migration effort: [Estimated effort for consumers]

**Proposed Changes**:
[Detailed description of changes]

**Alternatives Considered**:
[What other options were evaluated and why they were rejected]

**Timeline**:
- Proposed effective date: [Date]
- Deprecation date (if applicable): [Date]
```

### Step 3: Model Development

Follow SAMM (Semantic Aspect Meta Model) specifications:

```turtle
@prefix samm: <urn:samm:org.eclipse.esmf.samm:meta-model:2.1.0#> .
@prefix samm-c: <urn:samm:org.eclipse.esmf.samm:characteristic:2.1.0#> .
@prefix samm-e: <urn:samm:org.eclipse.esmf.samm:entity:2.1.0#> .
@prefix unit: <urn:samm:org.eclipse.esmf.samm:unit:2.1.0#> .
@prefix rdf: <http://www.w3.org/1999/02/22-rdf-syntax-ns#> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .
@prefix xsd: <http://www.w3.org/2001/XMLSchema#> .
@prefix : <urn:samm:org.eclipse.tractusx.mymodel:2.0.0#> .

:MyAspect a samm:Aspect ;
    samm:preferredName "My Aspect"@en ;
    samm:description "Description of what this aspect represents"@en ;
    samm:properties ( :property1 :property2 ) ;
    samm:operations ( ) ;
    samm:events ( ) .

:property1 a samm:Property ;
    samm:preferredName "Property 1"@en ;
    samm:description "Description of property 1"@en ;
    samm:characteristic samm-c:Text .
```

### Step 4: Documentation and Examples

Include:
- **README.md**: Overview and usage instructions
- **RELEASE_NOTES.md**: Changes in each version
- **Example JSON payloads**: Valid instance data
- **Migration guide**: For breaking changes

### Step 5: Validation

Validate the model using SAMM CLI:

```bash
# Install SAMM CLI
# Validate aspect model
samm aspect validate -i MyAspect.ttl

# Generate documentation
samm aspect generate docs -i MyAspect.ttl -o docs/

# Generate JSON schema
samm aspect generate json-schema -i MyAspect.ttl -o schema/
```

### Step 6: Review and Approval

The model undergoes review by:
1. **Technical review**: Data modeling experts check SAMM compliance
2. **Domain review**: Subject matter experts validate business semantics
3. **Architecture review**: Alignment with ecosystem architecture
4. **Approval**: Final approval by data modeling working group

### Step 7: Integration

Once approved:
- Merge to appropriate branch (main or version branch)
- Tag with version number
- Update dependent standards and documentation
- Communicate changes to ecosystem participants

## Examples

### Example 1: Adding an Optional Property (MINOR change)

:::tip Best Practice
Adding optional properties maintains backward compatibility.
:::

**Before (v1.0.0)**:
```turtle
:Vehicle a samm:Aspect ;
    samm:properties ( :vin :manufacturer ) .
```

**After (v1.1.0)**:
```turtle
:Vehicle a samm:Aspect ;
    samm:properties ( :vin :manufacturer :modelYear ) .

:modelYear a samm:Property ;
    samm:preferredName "Model Year"@en ;
    samm:description "The year the vehicle model was manufactured"@en ;
    samm:characteristic samm-c:Text ;
    samm:optional true .
```

**Why this is good**: The new property is optional, so existing data remains valid.

### Example 2: Breaking Change - New Major Version

:::danger Breaking Change
Changing property types requires a new major version.
:::

**Before (v1.5.0)**:
```turtle
:productionDate a samm:Property ;
    samm:preferredName "Production Date"@en ;
    samm:characteristic samm-c:Text .
```

**After (v2.0.0)**:
```turtle
:productionDate a samm:Property ;
    samm:preferredName "Production Date"@en ;
    samm:characteristic samm-c:Timestamp .
```

**Migration Guide**:
```markdown
## Migrating from v1.x to v2.0

### Breaking Change: productionDate format

**Old format** (v1.x): String "2024-01-15"
**New format** (v2.0): ISO 8601 timestamp "2024-01-15T10:30:00Z"

**Action required**: Update data transformation to convert strings to timestamps.
```

### Example 3: Anti-Pattern - Undocumented Change

:::danger What Not To Do
Making changes without documentation and proper versioning.
:::

```turtle
# Changed property name without version bump or documentation
:VIN a samm:Property ;  # Was :vin before
    samm:characteristic samm-c:Text .
```

**Why this is wrong**:
- Property rename is a breaking change
- No version bump (should be MAJOR)
- No documentation or migration guide
- Existing consumers will break

## FAQ

### Question 1: What is the difference between a semantic model change and an ontology change?

**Semantic models** (aspect models) define the structure of runtime data for specific use cases. They are technical data models.

**Ontologies** define the conceptual relationships and vocabulary used across the ecosystem. They are higher-level conceptual models.

For ontology changes, see [AGR-3.01](./agr-3-01.md).

### Question 2: How long must old versions be supported?

Major versions should be supported for at least:
- **12 months** after the next major version is released
- Long enough for all consumers to migrate

Deprecation notices must be published at least **6 months** before end-of-life.

### Question 3: Can I propose a model change if I'm not a committer?

Yes! Anyone can propose changes through GitHub issues. The review and approval process is open, though merge rights are restricted to maintainers.

### Question 4: What if my change doesn't fit the versioning rules?

Consult with the data modeling working group. There may be exceptional cases, but the default rules should be followed unless there's a strong justification.
